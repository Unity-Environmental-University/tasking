tasking

A loop,
not a list.

Tasks go in fast. The system surfaces them to you — in your terminal, in Claude — rather than waiting for you to remember to look.

The problem

Todo apps assume you'll go check them.

They assume you'll remember the app exists. That you'll open it. That you'll scan through it at the right moment. That you'll connect what you see to what you're working on right now.

For ADHD brains, that chain breaks at the first link.

tasking works the other way. You capture fast — one command, two seconds, no context switch. Then the system comes to you. Tasks appear when you open a terminal tab. Tasks flagged [c] surface in Claude's context before you even speak. The loop is always running.

Everything from one command.

Where tasks appear

The system comes to you.

You never have to open an app or remember to check. Click to see how each surface works.

🖥️
Terminal tab
Tasks appear when you open a new tab — global and current-repo local.
A Stop hook runs after every Claude response and every new terminal session. It shows today's tasks with a 5-minute cooldown so it's present but not noisy. Scoped tasks appear only in the right repo.
🤖
Claude context
Tasks tagged [c] surface before every Claude response — no prompting needed.
A UserPromptSubmit hook injects flagged tasks as additional context before Claude reads your message. Claude sees what you're tracking without you having to mention it. Use t c <text> to flag a task.
💬
Claude Desktop
The MCP server connects directly — Claude Desktop reads and writes tasks natively.
Add "tasking": {"url": "http://localhost:5055/mcp"} to your claude_desktop_config.json. Claude can then call add, complete, snooze, standup, and more as native tools — no copy-paste required.
🌿
Knowledge graph
Every lifecycle event writes an edge — adds, completes, snoozes, git commits.
Tasks trace into rhizome-alkahest as edges. Snooze patterns accumulate as signal — a task snoozed three times is asking something. Commits that reference #N link automatically via a global post-commit hook.
Scoping

Global or local to a repo.

Tasks are global by default — visible everywhere. Run t c from inside a git repo and the task automatically scopes to that repo, appearing only when you're working there.

t mv <id> toggles a task between local and global. Claude-flagged tasks default to local because they're almost always about your current context.

The task knows where it belongs. You don't have to manage it.

Get started

One script, then it runs forever.

Requires Node.js and PostgreSQL running locally. install.sh is idempotent — safe to run again if anything changes.

1

Clone and install

# clone the repo git clone https://github.com/Unity-Environmental-University/tasking cd tasking # one-shot installer: db, server, CLI, git hook, launchd bash install.sh
2

Connect Claude Desktop (optional)

Add the MCP server to your Claude Desktop config:

{ "mcpServers": { "tasking": { "url": "http://localhost:5055/mcp" } } }
3

Add Claude Code hooks (optional)

For tasks to surface in Claude Code sessions, add to ~/.claude/settings.json:

# UserPromptSubmit — injects [c] tasks before each response tasks=$(t claude_tasks 2>/dev/null) if [ -n "$tasks" ] && [ "$tasks" != 'No claude-tagged tasks.' ]; then printf '{"hookSpecificOutput": {"hookEventName": "UserPromptSubmit", "additionalContext": "Tasks flagged for your awareness:\n%s"}}' "$tasks" fi
4

Start capturing

# add a task t pick up thread on the mesh shader # add a task flagged for Claude (scoped to current repo) t c look at the particle lifetime issue # see today's tasks t
Key signals

What to watch for.

The system is more useful when you read its patterns, not just its tasks. Run t signal to surface them all at once.

A task snoozed three or more times is usually blocked, too big, or something you're avoiding. That's worth naming, not just rescheduling.

[c] tasks that accumulate without resolution mean Claude is aware of something you haven't addressed. Either act on them or cancel them — ambiguity has a cost.

The loop works best when it stays clean. Done means done. Snoozed means a specific date. Cancelled means it was never real.

Commits referencing #N write edges in rhizome automatically. Over time the graph shows you what you actually worked on versus what you planned to.
